Correction importante pour la Phase 8.6 — Ton éditorial et posture produit

Je veux une plateforme web très professionnelle, proche d’un vrai produit medtech/startup, moderne et convaincante.

Ne surchargez pas l’interface avec des avertissements partout.

La plateforme doit mentionner clairement qu’il s’agit d’un prototype académique, mais cette mention doit être sobre, contrôlée et élégante.

Objectif :
Donner une impression de produit mature, crédible et prêt pour démonstration, tout en restant honnête.

==================================================
1. RÈGLE ÉDITORIALE PRINCIPALE
==================================================

Ne pas répéter à chaque section :

- non certifié ;
- pas usage clinique ;
- pas données réelles ;
- pas validation clinique ;
- performances non transférables ;
- XAI ≠ causalité ;
- aucune dose ;
- aucune décision automatique.

Ces mentions sont importantes, mais elles doivent être regroupées dans des zones dédiées et formulées de manière professionnelle.

==================================================
2. OÙ AFFICHER LES LIMITES
==================================================

Afficher les limites principalement dans :

1. Une mention discrète dans le hero :
   Exemple :
   “Prototype académique — démonstration contrôlée”

2. Une section dédiée :
   “Cadre académique & limites”

3. Le footer :
   mention courte.

4. Les écrans critiques de l’app mobile :
   rappel sobre uniquement quand nécessaire, par exemple dans XAI/recommandations.

Ne pas mettre des warnings longs partout.

==================================================
3. TON À UTILISER
==================================================

Utiliser un ton produit/professionnel :

Préférer :

- “Prototype académique avancé”
- “Démonstration contrôlée”
- “Environnement de validation”
- “Données de démonstration”
- “Aide à la décision supervisée”
- “Validation humaine”
- “IA explicable”
- “Suivi diabète augmenté par IoMT”
- “Plateforme patient–clinicien”

Éviter de répéter partout :

- “non certifié”
- “pas usage clinique”
- “pas données réelles”
- “pas validation clinique”
- “pas de décision automatique”
- “aucune dose”

Ces termes restent dans la section limites, mais pas dans chaque card.

==================================================
4. HERO ATTENDU
==================================================

Le Hero doit ressembler à une vraie plateforme produit.

Exemple de direction :

Titre :
“MediAI Care”

Sous-titre :
“Plateforme IoMT et IA explicable pour le suivi personnalisé du diabète”

Texte :
“Un environnement patient–clinicien qui centralise les données, estime les risques glycémiques et rend les prédictions compréhensibles grâce à l’IA explicable.”

Mention discrète :
“Prototype académique — démonstration contrôlée”

CTA principal :
“Ouvrir l’application mobile”

CTA secondaire :
“Découvrir la plateforme”

Badges possibles :
- IoMT
- IA explicable
- Patient & clinicien
- Open-loop supervisé

Éviter dans le hero :
- gros avertissements rouges ;
- liste complète des limites ;
- “non certifié” répété en gros ;
- “pas usage clinique” comme élément dominant.

==================================================
5. SECTION LIMITES
==================================================

Créer une section dédiée, élégante, en bas de page :

Titre :
“Cadre académique”

Texte court :
“MediAI Care est présenté ici comme prototype académique avancé. Les données et scénarios de démonstration sont contrôlés. Toute évolution vers un usage réel nécessiterait validation clinique, conformité réglementaire et hébergement adapté.”

Puis liste sobre :

- Démonstration académique
- Données de démonstration
- Validation clinique non réalisée
- Décision humaine obligatoire
- Cadre réglementaire à formaliser avant usage réel

Ne pas utiliser un ton anxiogène.
Ne pas répéter cette liste partout ailleurs.

==================================================
6. APP MOBILE — POSTURE
==================================================

Dans l’app mobile, garder les rappels mais les rendre plus élégants.

Exemple :
“Environnement de démonstration académique”

Au lieu de :
“Données 100 % simulées. Non certifié. Pas usage clinique. Aucune décision…”

Sur les écrans sensibles comme XAI/recommandations, garder un rappel court :

“Suggestion supervisée — validation clinicien requise”

ou :

“Explication du modèle, à interpréter avec prudence”

Mais ne pas alourdir tous les écrans.

==================================================
7. DESIGN ATTENDU
==================================================

La plateforme doit être plus proche d’un vrai produit que d’un rapport de conformité.

Inspiration possible :
- Glooko XT pour la structure ;
- Dexcom pour la clarté produit ;
- mySugr pour l’accessibilité ;
- Tidepool pour l’open data / clarté ;
- Livongo pour le positionnement patient-clinicien.

Mais ne pas copier :
- logos ;
- textes ;
- images ;
- certifications ;
- claims cliniques.

==================================================
8. CE QUI DOIT DOMINER VISUELLEMENT
==================================================

Ce qui doit dominer :

- valeur produit ;
- app mobile ;
- expérience patient ;
- expérience clinicien ;
- IA explicable ;
- centralisation données ;
- workflow supervisé ;
- design moderne ;
- crédibilité.

Ce qui doit rester présent mais discret :

- statut académique ;
- limites ;
- non usage réel ;
- conformité future.

==================================================
9. À MODIFIER DANS LE PROMPT PHASE 8.6
==================================================

Dans la Phase 8.6, remplacer toute formulation qui demande d’afficher partout :

- non certifié ;
- pas usage clinique ;
- pas données réelles ;
- performances non transférables ;
- aucune dose ;
- aucune décision automatique.

Par :

“Afficher une mention sobre dans le hero, une section dédiée Cadre académique & limites, et un rappel court dans le footer. Ne pas répéter les limites dans chaque section.”

==================================================
10. CRITÈRE D’ACCEPTATION SPÉCIFIQUE
==================================================

La plateforme sera validée si :

- elle donne une impression professionnelle et mature ;
- elle ne ressemble pas à une page d’avertissement ;
- elle garde une mention claire du cadre académique ;
- elle présente fortement la valeur produit ;
- elle montre clairement que l’application mobile est le vrai produit ;
- les limites sont accessibles mais non envahissantes ;
- aucun claim clinique ou certification n’est inventé.

Conclusion :
Refondre la plateforme avec une posture produit premium, proche d’un vrai pré-MVP medtech, en gardant les limites dans une section dédiée et non répétées partout.

## Reformulation de la logique

Ce qu’il faut viser :

```text
80 % produit / valeur / design / expérience
20 % cadre académique / limites / transparence
```

Pas :

```text
20 % produit
80 % avertissements
